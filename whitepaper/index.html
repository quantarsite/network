Section 0

Executive Summary

A post-quantum transition is not only a cryptographic replacement problem. It is a systems problem involving policy, account lifecycle, upgrade discipline, operational recovery, and interoperability across different assurance levels.

Quantar Network addresses this problem at the most fundamental layer: the consensus protocol itself. Rather than retrofitting post-quantum signatures onto a chain that launched with classical cryptography, Quantar enforces dual post-quantum authentication from genesis through the HYDRA-X protocol.

HYDRA-X implements AND-composed dual signatures using ML-DSA-87 (FIPS 204) and SLH-DSA-SHAKE-256f / SPHINCS+ (FIPS 205). Both components are bound to a shared transcript that carries message content, key references, domain context, and migration state. Verification requires both components to pass — a transaction is valid only when both the lattice-based and hash-based signatures verify against the composed transcript.

This document collects three technical perspectives that emerged from our protocol design: conjunctive hybrid signing and the AND-composition model, component binding with explicit transcript construction, and controlled credential migration with monotonic policy enforcement.

The main recommendation is conservative. Hybrid designs should be treated as policy-bearing objects. Context, key identifiers, account state, and migration stage should be explicit in what is authenticated, and downgrade paths should be narrow, auditable, and exceptional.

Section 1

Why Hybrid Signatures Matter

A hybrid signature combines two independent authentication components, typically one conventional or post-quantum signature and one structurally different post-quantum signature. The transition rationale is straightforward: an operator may want hedging against the possibility that one post-quantum family is broken while the other survives.

In Quantar's case, the hybrid is not classical + post-quantum. It is lattice-based + hash-based, both post-quantum. ML-DSA-87 provides compact signatures with strong performance characteristics rooted in lattice hardness. SLH-DSA / SPHINCS+ provides conservative hash-based security with minimal structural assumptions. The AND-composition means an attacker must break both families simultaneously to forge a valid transaction.

For Quantar, the practical value of hybrid authentication is not merely redundancy. It is a way to introduce migration discipline from genesis: every account starts in a post-quantum state, and the protocol never needs to manage the messy transition from classical to quantum-resistant credentials that retrofit designs require.

Section 2

Threat Model for Secure Transition

A transition design should distinguish between primitive-level failures and policy-level failures. Primitive-level failures include forgery of a component signature or collision-like ambiguity in a binding mechanism. Policy-level failures include unauthorized downgrade, incorrect account-state transitions, and misuse of privileged recovery pathways.

The most important near-term threat is not necessarily an immediate quantum break. It is the accumulation of technical debt around upgrade handling: unclear state transitions, weak component binding, emergency authority that quietly bypasses policy, and ambiguous verifier logic. Chains that launched with ECDSA or EdDSA and plan to migrate later carry this debt structurally. Quantar's genesis-native approach eliminates this entire class of risk.

Cryptographic Compromise Cross-Component Ambiguity Operational Rollback Governance Misuse
Forgery or break of a component algorithm (lattice or hash-based). Recombining valid-looking pieces from different signing events. Re-activating weaker credentials after a stronger one is registered. Abuse of privileged update authority or emergency procedures.
Section 3

AND-Composition and Component Binding

A naive hybrid signature can be represented as an ordered pair of independently valid signatures. That representation captures conjunctive verification but does not by itself say whether the two components are meant to belong to the same composed signing event. This matters whenever verifiers care about domain, profile, state, or key-pair identity rather than the raw payload alone.

HYDRA-X addresses this through AND-composition with explicit component binding. The construction is formalized in our research on AND-composed signatures, which introduces the CB-EUF-CMA (Component-Bound Existential Unforgeability under Chosen Message Attack) security experiment. The core property: a composed signature is valid only when both components authenticate the same bound transcript, and no partial recombination of components from different signing events can produce a valid composed signature.

Both components sign a shared, deterministic transcript rather than the raw payload alone. The transcript carries the message, the context in which verification is expected to occur, and explicit references that bind the two components together.

HYDRA-X Binding Transcript

μ = H( domain || account_id || profile_id || pk_mldsa_ref || pk_slhdsa_ref || migration_state || payload_digest )

Where H is a collision-resistant hash, domain is the HYDRA-X protocol domain tag, pk_mldsa_ref and pk_slhdsa_ref are stable references to the ML-DSA-87 and SLH-DSA public keys respectively, and migration_state encodes the current account assurance profile.

This is not merely a formatting convention. The transcript construction ensures that the composed signature carries an unambiguous commitment to the identity, policy state, and algorithmic context of the signing event. Stripping or substituting any component invalidates the binding.

Section 4

Controlled Migration and Upgrade Policies

Credential migration should be modeled as a state machine rather than a one-time operational script. Each account or identity should have a declared authentication profile, and policy should specify which transitions are ordinary, which are privileged, and which are forbidden after upgrade.

Because Quantar enforces HYDRA-X from genesis, the migration model is simpler than in retrofit chains: every account begins in the strongest assurance state. The state machine governs forward evolution — for example, algorithm agility when new NIST standards emerge — rather than the far riskier backward migration from classical to post-quantum.

Section 5

HYDRA-X Protocol Specifics

HYDRA-X is the dual post-quantum signature protocol enforced at the consensus layer of Quantar Network. It is not an application-layer wrapper or an optional module. Every block, every transaction, and every validator attestation must carry a valid HYDRA-X composed signature to be accepted by the network.

Component Algorithms

ComponentStandardSecurity LevelRole in HYDRA-X
ML-DSA-87FIPS 204NIST Level 5Primary: compact, fast, lattice-based
SLH-DSA-SHAKE-256fFIPS 205NIST Level 5Secondary: conservative, hash-based, minimal assumptions

The choice of ML-DSA-87 and SLH-DSA is deliberate. These represent structurally different hardness assumptions: lattice problems (Module-LWE / Module-SIS) for ML-DSA and hash function security for SLH-DSA. A breakthrough against lattice-based cryptography would not affect the hash-based component, and vice versa. The AND-composition guarantees that the composed scheme is at least as strong as the stronger of the two components.

Section 6

Practical Hybrid Design Notes

✓
Domain Separation Dedicated HYDRA-X domain tag in transcript derivation. Not reused from other protocol layers.
✓
Key References Stable references to both ML-DSA-87 and SLH-DSA public keys bound in every signature.
✓
Policy State Migration stage and assurance profile included in signed transcript.
✓
Recovery Scope Exceptional authority distinguishable from ordinary account authorization. Time-locked.
✓
Consensus Enforcement Dual verification mandatory at block validation. No single-component bypass path.

Treat algorithm identifiers as first-class authenticated data. Many deployment mistakes come from assuming that the verifier can infer the intended algorithm profile from context. In HYDRA-X, the profile identifier is part of the signed transcript.

Avoid overloading a single key slot with different assurance meanings. In HYDRA-X, the ML-DSA-87 key and the SLH-DSA key are represented as distinct objects with distinct policy roles.

Use narrow recovery authorities. Emergency recovery is operationally necessary, but it should not become a hidden universal bypass. In Quantar, recovery operations require separate authorization, time-locks, and are logged at the consensus layer.

Prefer deterministic object formats for what gets signed. Ambiguous serialization, optional metadata with inconsistent defaults, or unclear domain separation can undermine the operational value of a carefully designed bound profile.

Section 7

Genesis-Native vs. Retrofit

Most existing blockchain networks were designed around classical elliptic-curve cryptography (ECDSA, EdDSA). Migrating these networks to post-quantum signatures requires either a hard fork that replaces the authentication scheme or a wrapper layer that adds post-quantum signatures alongside existing ones. Both approaches carry structural risk.

Retrofit Approach

✕High migration complexity — must manage classical + PQC coexistence
✕Accounts may silently fall back to classical-only during transition
✕Verifiers must handle multiple authentication paths
✕Technical debt accumulates: legacy code, emergency bypasses, edge cases

Genesis-Native (Quantar)

✓No migration — all accounts start with dual PQC from block one
✓Downgrade eliminated by design: no classical credentials exist
✓Single verification path (HYDRA-X) from genesis — no ambiguity
✓Minimal debt: protocol designed for post-quantum from the start
Section 8

Limits and Open Problems

This whitepaper is deliberately conservative. It does not claim a new cryptographic primitive, a full proof framework for all hybrid deployments, or a complete account-migration protocol for every possible network topology.

Several open problems remain important for future work: more formal treatment of multi-user hybrid state in decentralized settings, explicit models for delegated recovery authority under post-quantum assumptions, analysis of algorithm agility under long-lived account semantics (particularly as NIST standards evolve), and sharper proof frameworks for component binding across protocol layers.

For practical engineering, the takeaway remains clear. Hybrid transition should be treated as a disciplined protocol and policy design problem. The more explicitly the system models context, state, and authority, the less likely it is to confuse surface-level dual signing with genuine upgrade security.
